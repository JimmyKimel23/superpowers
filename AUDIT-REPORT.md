## Находки

### [CRITICAL] ID-001: API-ключ доступен любому посетителю страницы
**Что не так:** Поле `apiKey` в блоке CONFIG хранится прямо в HTML-файле. Любой, кто откроет страницу, может посмотреть исходный код (Ctrl+U) и скопировать ключ.
**Чем опасно:** Злоумышленник может использовать украденный ключ для отправки запросов к API за ваш счёт — это прямые финансовые потери и возможная утечка данных через API.
**Строки:** 415–416
**Статус:** DEFERRED
**Что сделано:** —

---

### [CRITICAL] ID-002: Системный промпт читается через браузер
**Что не так:** Системный промпт (инструкции для модели) хранится в скрытом HTML-элементе `<textarea>`. Элемент скрыт визуально, но доступен через «Просмотр кода страницы» или консоль браузера.
**Чем опасно:** Зная системный промпт, пользователь может подобрать формулировки, чтобы обойти ограничения модели — например, заставить её отвечать на вопросы, которые промпт запрещает.
**Строки:** 440–442
**Статус:** FIXED
**Что сделано:** Промпт перенесён из DOM-элемента `<textarea>` в JavaScript-переменную внутри блока `<script>`. Это не делает его полностью невидимым (исходный код всё равно можно посмотреть), но убирает из DOM — промпт больше нельзя прочитать одной командой в консоли. HTML-комментарий с инструкцией по вставке промпта обновлён.

---

### [HIGH] ID-003: Нет таймаута при запросе к серверу
**Что не так:** Запрос к API (`fetch`) отправляется без ограничения по времени. Если сервер «завис» и не отвечает, виджет ждёт ответа бесконечно.
**Чем опасно:** Кнопка «Отправить» остаётся заблокированной, индикатор «печатает...» крутится без конца. Пользователю приходится перезагружать страницу и терять весь диалог.
**Строки:** 632–641
**Статус:** FIXED
**Что сделано:** Добавлен таймаут 30 секунд через `AbortController`. Если сервер не ответил за 30 секунд — запрос отменяется, пользователь видит понятное сообщение об ошибке, кнопка разблокируется.

---

### [HIGH] ID-004: История диалога растёт без ограничений
**Что не так:** Каждое сообщение (и пользователя, и модели) добавляется в массив и при следующем запросе отправляется целиком в API. Массив никогда не обрезается.
**Чем опасно:** После 50–100 сообщений тело запроса становится настолько большим, что API отклоняет его (ошибка 413 или лимит токенов). С этого момента все новые сообщения перестают работать.
**Строки:** 482, 615, 622, 657
**Статус:** FIXED
**Что сделано:** Добавлено ограничение: в API отправляются только последние 50 сообщений. Старые сообщения по-прежнему видны в окне чата, но не включаются в запрос. Это предотвращает превышение лимитов API.

---

### [MEDIUM] ID-005: Нет ограничения длины ввода
**Что не так:** Поле ввода сообщения не имеет ограничения по количеству символов. Пользователь может вставить текст любой длины — хоть целую книгу.
**Чем опасно:** Очень длинное сообщение создаст гигантский запрос к API, который будет отклонён или приведёт к замедлению браузера. Также это может использоваться для намеренной перегрузки API.
**Строки:** 471
**Статус:** FIXED
**Что сделано:** Добавлен атрибут `maxlength="4000"` на поле ввода. 4000 символов — это примерно 1–2 страницы текста, достаточно для любого разумного вопроса.

---

### [MEDIUM] ID-006: Нумерованные списки отображаются как маркированные
**Что не так:** Markdown-парсер обрабатывает маркированные списки (`- текст`) и оборачивает их в `<ul>`. Но нумерованные списки (`1. текст`) обрабатываются после, и их элементы не оборачиваются в `<ol>` — они остаются «голыми» `<li>` без родительского тега.
**Чем опасно:** Нумерованные списки в ответах модели отображаются некорректно — без нумерации, без отступов, с нарушением вёрстки.
**Строки:** 587–589
**Статус:** FIXED
**Что сделано:** Нумерованные списки теперь обрабатываются отдельным regex и оборачиваются в `<ol>`. Добавлена очистка `<p>`-обёрток для `<ol>` аналогично `<ul>`.

---

### [MEDIUM] ID-007: Тема не сохраняется при перезагрузке
**Что не так:** Когда пользователь переключает тему (светлая/тёмная), выбор нигде не сохраняется. При перезагрузке страницы всегда включается тёмная тема.
**Чем опасно:** Это не опасно, но раздражает. Если пользователю удобнее светлая тема, ему приходится переключать её каждый раз при входе.
**Строки:** 496–499
**Статус:** FIXED
**Что сделано:** Выбор темы сохраняется в `localStorage`. При загрузке страницы тема восстанавливается автоматически. Иконка (солнце/луна) тоже обновляется при восстановлении.

---

### [LOW] ID-008: Нет поддержки экранных читалок (accessibility)
**Что не так:** Контейнер сообщений не имеет атрибутов `role="log"` и `aria-live`, кнопки не имеют `aria-label`. Экранные читалки (программы для слабовидящих) не могут нормально озвучивать содержимое чата.
**Чем опасно:** Слабовидящие пользователи не смогут полноценно пользоваться виджетом. Также это может нарушать требования по доступности (WCAG).
**Строки:** 461, 471–472
**Статус:** FIXED
**Что сделано:** Добавлены `role="log"` и `aria-live="polite"` на контейнер сообщений. Добавлен `aria-label` на кнопку отправки и поле ввода.

---

### [LOW] ID-009: Все функции и переменные в глобальной области видимости
**Что не так:** Все функции (`sendMessage`, `clearChat`, `toggleTheme`) и переменные (`conversationHistory`, `isLoading`, `CONFIG`) доступны глобально. Если на той же странице работают другие скрипты, они могут случайно перезаписать эти имена.
**Чем опасно:** Конфликт имён может сломать виджет без видимой причины. На практике, пока виджет используется как отдельная страница — проблемы нет.
**Строки:** 482–684
**Статус:** DEFERRED
**Что сделано:** —

## Не исправлено (DEFERRED / WONTFIX)

### ID-001: API-ключ в клиентском коде
Это ограничение архитектуры: виджет работает целиком в браузере, без серверной части. Спрятать ключ в таком виджете невозможно — для этого нужен серверный прокси (промежуточный сервер, который хранит ключ у себя и пересылает запросы к API). Рекомендация: если API требует ключ, разверните простой прокси-сервер и укажите его адрес в CONFIG.endpoint. Ключ тогда будет храниться только на сервере.

### ID-009: Глобальные переменные
Оборачивание всего кода в замыкание (IIFE) — это существенный рефакторинг, который затронет все функции. Пока виджет используется как отдельная HTML-страница (а не встраивается в другой сайт), конфликтов не будет. Если в будущем виджет будет встраиваться в другие страницы — стоит обернуть код в модуль.
